# Step 4：バックプレッシャーと遅いサブスクライバー対策（前提知識）

Step 3 では `sub.ch <- message` がブロッキングのままです。処理が遅いサブスクライバーのチャネルが満杯になると、`Publish` 全体が止まり、速いサブスクライバーへの配信も遅延します。Step 4 ではこの問題を「ドロップ戦略」で解決します。

---

## 1. バックプレッシャーとは

上流（Publisher）の速度が下流（Subscriber）の処理速度を超えるとき、メッセージが溜まっていく圧力のことです。

```
Publisher: 1000 msg/sec
Subscriber: 100 msg/sec  ← 10倍遅い

→ 1秒ごとに 900 件のメッセージが溜まる
→ チャネルのバッファが枯渇
→ Publisher がブロック
```

---

## 2. 3つの対処戦略

| 戦略 | 概要 | 向いているケース |
|---|---|---|
| ブロック | 遅いサブスクライバーが追いつくまで待つ | データを1件も落とせない場合 |
| ドロップ（今回） | 満杯なら新しいメッセージを捨てる | リアルタイム性が重要で多少の欠損を許せる場合 |
| 切断 | 遅すぎるサブスクライバーを強制切断する | SLAが厳しいシステムで他のサブスクライバーを守る場合 |

Pub/Subではリアルタイム配信が目的のケースが多く、**ドロップ戦略**が最も一般的です。

---

## 3. `select` の `default` 節によるノンブロッキング送信

Goのチャネルへの送信は通常ブロッキングです。`select` の `default` を使うと、即座に送信できない場合にブロックせず `default` ブロックに進みます。

```go
select {
case sub.ch <- message:
    // 送信成功
default:
    // チャネルが満杯: ドロップ
    atomic.AddInt64(&sub.dropped, 1)
}
```

これにより `Publish` のループは各サブスクライバーに対して**常に即座に進む**ことが保証されます。

---

## 4. ドロップカウンターとログ

どのサブスクライバーが何件ドロップしたかを記録します。`sync/atomic` でgoroutine-safeにインクリメントし、定期的にログに出します。

```
[WARN] subscriber 127.0.0.1:54321 topic=news dropped=42 messages
```

このログを見てサブスクライバー側のチューニングが必要かどうか判断できます。

---

## 📌 まとめ：Step 4 の変更点

`Publish` 内の `sub.ch <- message` を以下に置き換えるだけです。

```go
// Before（ブロッキング）
subscription.ch <- message

// After（ノンブロッキング・ドロップ）
select {
case subscription.ch <- message:
default:
    atomic.AddInt64(&sub.dropped, 1)
}
```

それ以外のコードは Step 3 から変更不要です。
