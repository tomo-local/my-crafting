# Step 6：Replayバッファ（前提知識）

Step 5 まではサブスクライブした時点以降のメッセージしか受け取れませんでした。Step 6 では各トピックの直近N件を保持するリングバッファを実装し、新規サブスクライバーに過去メッセージを即座に配信します。

---

## 1. なぜReplayバッファが必要か

Pub/Subでは「最初にサブスクライバーを起動してからパブリッシャーを起動する」順序が必要です。しかし実際のシステムでは順序を保証できないことがあります。

```
パブリッシャーが起動してメッセージを送り始める
  ↓
サブスクライバーが少し遅れて起動する
  ↓
接続前のメッセージは受け取れない（通常の動作）

Replayがあれば:
  ↓
サブスクライブ時に過去N件が自動的に送られる
```

---

## 2. リングバッファ（循環バッファ）

固定サイズの配列を「輪」として使うデータ構造です。古いメッセージを自動的に上書きするため、常に「最新N件」が保持されます。

```
容量 4 のリングバッファ:

初期状態:  [_, _, _, _]  head=0, count=0

msg1追加:  [A, _, _, _]  head=0, count=1
msg2追加:  [A, B, _, _]  head=0, count=2
msg3追加:  [A, B, C, _]  head=0, count=3
msg4追加:  [A, B, C, D]  head=0, count=4

msg5追加（上書き）:
           [E, B, C, D]  head=1, count=4  ← Aが上書きされ、headが1に進む

読み出し順: B → C → D → E  （head から count 件）
```

インデックス計算: `(head + i) % capacity`

---

## 3. `SUB` コマンドへのReplay件数指定

Replayの件数はサブスクライバーが選択できるようにします。

```
SUB <topic> [replay_count]

例:
SUB news       → Replay なし（通常動作）
SUB news 5     → 過去5件をReplayしてから通常配信
SUB news -1    → 全件Replay（バッファの全容量）
```

---

## 4. Replay中の排他制御

Replayはサブスクライブ処理の中で行います。Replay中に新しいメッセージが `Publish` される可能性があります。

```
Replay 開始
  → バッファを読んで過去メッセージを送る  ← この間に PUB が来る可能性
Replay 終了
  → 通常の配信チャネルに切り替える        ← Replay 後のメッセージが欠落する危険
```

**対策**: Replayと新規メッセージをシームレスにつなぐため、以下の順序で処理します。

1. `b.mu.Lock()` でブローカーをロックする
2. バッファの内容をスナップショットする
3. サブスクライバーをリストに追加する（以降の Publish はこのサブスクライバーにも届く）
4. `b.mu.Unlock()`
5. スナップショットを ch に送る（Replay）
6. 通常配信はチャネル経由で自動的に届く

この順序により、ロック中に届いたメッセージはサブスクライバーのチャネルに入り、Replayメッセージの後に届きます。

---

## 📌 まとめ：Step 6 のデータ構造

**トピックごとに `RingBuffer` を持つ:**

```go
type TopicData struct {
    subscribers []*Subscriber
    buffer      *RingBuffer
}

// Broker
topics map[string]*TopicData
```

**`Publish` 時に:**
1. `buffer.Push(message)` でメッセージを記録する
2. サブスクライバーにファンアウトする

**`Subscribe` 時に:**
1. ロック中にバッファのスナップショットを取る
2. サブスクライバーをリストに追加する
3. ロック解除後にスナップショットを ch に送る
