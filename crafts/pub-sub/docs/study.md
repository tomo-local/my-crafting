# Pub/Subブローカー自作に必要な知識の整理

Pub/Subブローカーの自作は、大きく分けて**「Pub/Subパターンの設計思想」「メッセージングプロトコルの設計」「Goの並行処理とチャネル」「バックプレッシャーと配信保証」**の4つの知識が必要です。

---

## 1. Pub/Subパターンとは何か

### 📬 直接通信との違い

通常のクライアント/サーバー通信では、送信者が受信者を直接知っています。

```
直接通信（Direct）:
  Producer ────────────────> Consumer
  （送信先を知っていないと送れない）

Pub/Sub:
  Publisher ──> [Broker: topic "news"] ──> Subscriber A
                                       ──> Subscriber B
                                       ──> Subscriber C
  （送信者は誰が受け取るかを知らない）
```

この「間接性」が Pub/Sub の本質です。送信者（Publisher）と受信者（Subscriber）は**互いの存在を知らない**まま通信できます。

### 🏗️ ブローカーの役割

ブローカーは「郵便局」のようなものです。

| 役割 | 内容 |
|---|---|
| トピック管理 | `"news"` や `"events.sports"` などのトピックを管理する |
| サブスクリプション管理 | どのクライアントがどのトピックを購読しているかを記録する |
| ファンアウト | 1件のメッセージを、そのトピックの全サブスクライバーに配信する |
| バッファリング | サブスクライバーが遅い場合にメッセージを一時保持する |

## 2. プロトコルの設計

### 📝 行ベースのテキストプロトコル

このクラフトでは NATS を参考にした、シンプルな行ベースプロトコルを使います。1コマンド = 1行（`\r\n` 終端）です。

| コマンド | 送信方向 | 形式 | 意味 |
|---|---|---|---|
| `SUB` | クライアント → ブローカー | `SUB <topic>` | トピックを購読する |
| `UNSUB` | クライアント → ブローカー | `UNSUB <topic>` | 購読を解除する |
| `PUB` | クライアント → ブローカー | `PUB <topic> <message>` | メッセージを発行する |
| `MSG` | ブローカー → クライアント | `MSG <topic> <message>` | メッセージを配信する |
| `+OK` | ブローカー → クライアント | `+OK` | コマンド受信確認 |
| `-ERR` | ブローカー → クライアント | `-ERR <reason>` | エラー通知 |

### 📥 パース方針

1行を `strings.Fields` で分割し、先頭がコマンド名、2番目以降が引数です。

```
"PUB news Hello World" 
  → fields[0] = "PUB"
  → fields[1] = "news"
  → fields[2:] = ["Hello", "World"] → strings.Join で "Hello World"
```

## 3. ファンアウトとGoのチャネル

### 📡 チャネルによる非同期配信

各サブスクライバーは専用の `chan string` を持ちます。ブローカーがメッセージを受け取ったとき、そのトピックのサブスクライバー全員のチャネルに書き込みます。

```
Broker.Publish("news", "Hello")
    ↓
for _, sub := range broker.subscribers["news"] {
    sub.ch <- "Hello"    ← 各サブスクライバーのチャネルに書き込む
}
```

各サブスクライバーは別goroutineでチャネルを読み出し、TCPコネクションに書き込みます。

```
goroutine per subscriber:
    for msg := range sub.ch {
        fmt.Fprintf(conn, "MSG %s %s\r\n", topic, msg)
    }
```

### 🔀 ファンアウトの並列性

複数のサブスクライバーへの配信を直列にすると、1人のサブスクライバーが遅い場合に全員への配信が遅くなります。チャネルへの書き込みは非同期なので、**ブローカーはチャネルに入れるだけ**で次のメッセージ処理に進めます。実際のTCPへの書き込みは各サブスクライバーのgoroutineが担います。

## 4. サブスクライバーリストの排他制御

### 🔒 なぜ排他制御が必要か

以下の操作が同時に発生する可能性があります。

```
goroutine A（Publish）: subscribers["news"] を読み取ってファンアウト
goroutine B（新規SUB）: subscribers["news"] に新サブスクライバーを追加
goroutine C（切断処理）: subscribers["news"] からサブスクライバーを削除
```

`map` と `slice` はGoではgoroutine-safeではないため、同時アクセスはデータ競合を引き起こします。

### 🛡️ `sync.RWMutex` の使い分け

Publish（読み取り）は頻繁に起き、SUB/UNSUB/切断（書き込み）は相対的に少ない。

```
読み取り（Publishでサブスクライバーリストを参照）: RLock / RUnlock
書き込み（SUB/UNSUB/切断でリストを変更）:          Lock / Unlock
```

## 5. バックプレッシャーの概念

### ⚠️ チャネルが満杯になると何が起きるか

バッファ付きチャネル `make(chan string, N)` は N 件まで溜められます。N 件を超えた時点で書き込みがブロックします。

```
サブスクライバーのチャネル（容量 10）:
[msg1, msg2, ..., msg10]  ← 満杯

次の Publish:
    sub.ch <- msg11   ← ここでブロック！
    （このゴルーチンが止まるとブローカー全体が止まる）
```

### 💧 ドロップ戦略（non-blocking send）

満杯の場合にブロックせず、メッセージを捨てる実装が「ドロップ戦略」です。

```go
select {
case sub.ch <- msg:
    // 送信成功
default:
    // 満杯なので捨てる（ドロップカウンターをインクリメント）
}
```

`select` の `default` 節は「チャネルへの書き込みが即座に完了しなければ実行する」分岐です。これによりブロッキングを完全に回避できます。

## 6. ワイルドカードマッチングの仕組み

### 🎯 トピックの階層構造

トピック名は `.` で区切られた階層構造を持ちます。

```
"news.sports.football"
 └── セグメント1: "news"
      └── セグメント2: "sports"
           └── セグメント3: "football"
```

### 🃏 ワイルドカードの意味

| パターン | 意味 | マッチ例 |
|---|---|---|
| `*` | 任意の1セグメント | `news.*` → `news.sports` ○、`news.sports.football` ✗ |
| `>` | 残り全セグメント（1以上） | `news.>` → `news.sports` ○、`news.sports.football` ○ |

### 🔍 マッチングアルゴリズム

トピック名とパターンをどちらも `.` で `strings.Split` して、セグメントを1つずつ比較します。

```
パターン: ["news", "*"]
トピック: ["news", "sports"]

比較:
  news == news  → OK
  * == sports   → * は何でもマッチ → OK
  両方終了       → マッチ成功 ✓

パターン: ["news", "*"]
トピック: ["news", "sports", "football"]

比較:
  news == news  → OK
  * == sports   → OK
  パターン終了、トピックが余る → マッチ失敗 ✗
```

`>` は「残り全部」なので、それ以降のトピックセグメントを全て受け入れて終了します。

## 7. 【実装上の罠】Pub/Subで必ずハマるポイント

### 🛑 goroutineリーク

サブスクライバーが切断してもgoroutineが残るケースが頻発します。よくあるパターン：

```
配信goroutine: for msg := range sub.ch { ... }
切断処理: sub.ch への書き込み停止（でも close(sub.ch) を忘れる）
→ goroutine が for range を抜けられず永久にブロック
```

**対策**: サブスクライバーをリストから削除するとき、必ず `close(sub.ch)` を呼ぶ。`for range` はチャネルが閉じると終了します。

### 🛑 同じサブスクライバーへの重複配信

1つのクライアントが同じトピックに2回 `SUB` すると、リストに2つ追加されてメッセージが2回届きます。

**対策**: `SUB` 処理時に既登録かチェックするか、`map[string]*Subscriber` で重複を防ぐ。

### 🛑 `close` 済みチャネルへの書き込みパニック

```go
close(sub.ch)  // 切断処理
// ...
sub.ch <- msg  // panic: send on closed channel
```

Publish と切断処理が競合するとパニックが発生します。`recover` で握りつぶすより、チャネルが閉じられているかフラグで管理する方が安全です。
