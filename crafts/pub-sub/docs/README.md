# ゼロから構築するPub/Subブローカー自作・全体学習プロセス

標準ライブラリのチャネルを組み合わせながら、TCP上に独自のメッセージングプロトコルを実装し、Pub/Subブローカーの本質（トピックルーティング・ファンアウト・バックプレッシャー・ワイルドカード）を段階的に理解していく手順です。

---

## 全体ロードマップ

| Step | テーマ | ゴール |
|------|--------|--------|
| 1 | 最小Pub/Sub | `nc` でSUB/PUBコマンドを打つとメッセージが届くこと |
| 2 | ファンアウト | 同じトピックのサブスクライバー全員にメッセージが届くこと |
| 3 | Subscribe/Unsubscribeのライフサイクル | 切断・UNSUB後はメッセージが届かなくなること |
| 4 | バックプレッシャーと遅いサブスクライバー対策 | 遅いサブスクライバーがいても速いサブスクライバーへの配信が止まらないこと |
| 5 | ワイルドカードサブスクリプション | `news.*` で `news.sports` と `news.tech` 両方が届くこと |
| 6 | Replayバッファ | 新規サブスクライブ時に過去N件のメッセージが届くこと |

## 各ステップの詳細

### Step 1: 最小Pub/Sub

Pub/Subの本質は「パブリッシャーとサブスクライバーがトピック名を介して間接的に通信する」ことです。まず1トピック・1サブスクライバーの最小形を作ります。

**学習内容**
- Pub/Subパターンの設計思想（なぜ直接通信でないのか）
- NATSインスパイアの行ベースTCPプロトコル（`SUB`, `PUB`, `MSG`）
- Goのgoroutineとチャネルによる非同期配信の基礎

**実験ゴール**

```bash
# ターミナル A: ブローカー起動
go run main.go -port 4222

# ターミナル B: サブスクライバー
nc localhost 4222
SUB news
# → +OK

# ターミナル C: パブリッシャー
nc localhost 4222
PUB news Hello World
# → +OK

# ターミナル B に表示されること
MSG news Hello World
```

---

### Step 2: ファンアウト

同一トピックに複数のサブスクライバーが存在するとき、全員に配信します。これがPub/Subの核心機能です。

**学習内容**
- トピックごとのサブスクライバーリスト管理（`map[string][]*Subscriber`）
- ファンアウト: 1メッセージを複数チャネルにコピーして送る
- goroutine per subscriber による並列配信

**実験ゴール**

```bash
# サブスクライバーを2つ起動してSUB news
# パブリッシャーが1回PUBすると両方に届くこと

# ターミナルB: SUB news
# ターミナルC: SUB news
# ターミナルD: PUB news Hello
# → B と C 両方に MSG news Hello が届くこと
```

---

### Step 3: Subscribe/Unsubscribeのライフサイクル

サブスクライバーが切断したりUNSUBしたりしたとき、そのサブスクライバーへの配信を止めます。

**学習内容**
- クライアントの切断検知（`io.EOF` のハンドリング）
- `UNSUB <topic>` コマンドの実装
- 接続ごとのサブスクリプション状態の管理
- `sync.RWMutex` によるサブスクライバーリストの排他制御

**実験ゴール**

```bash
# ターミナルB: SUB news してから UNSUB news
UNSUB news
# → その後 PUB news されても B には届かないこと

# ターミナルB を Ctrl+C で切断
# → 切断後に PUB されても B には届かないこと（ブローカーがクラッシュしないこと）
```

---

### Step 4: バックプレッシャーと遅いサブスクライバー対策

サブスクライバーの処理が遅いと、メッセージが配信チャネルに溜まり続けます。放置するとメモリが枯渇したり、速いサブスクライバーへの配信まで影響が出ます。

**学習内容**
- Goのバッファ付きチャネルとブロッキング挙動
- ドロップ戦略: チャネルが満杯なら新しいメッセージを捨てる
- ドロップログ: いつ何件落としたかを記録する
- ブロック戦略との比較（トレードオフ）

**実験ゴール**

```bash
# 処理が遅いサブスクライバーを作る（ブローカーとは別のプログラムで）
go run subscriber/main.go -delay 500ms -topic news

# 速いサブスクライバーも起動
go run subscriber/main.go -delay 0ms -topic news

# 大量メッセージをパブリッシュ
go run publisher/main.go -topic news -count 1000 -interval 1ms

# → 速いサブスクライバーはほぼ全件受け取れること
# → 遅いサブスクライバーはドロップログが出ること（ブローカーがハングしないこと）
```

---

### Step 5: ワイルドカードサブスクリプション

`news.*` で `news.sports` にも `news.tech` にもマッチするトピックパターンを実装します。

**学習内容**
- トピック名の階層構造（`.` 区切り）
- ワイルドカードの種類:
  - `*`: 1セグメントにマッチ（`news.*` → `news.sports` ○、`news.sports.football` ✗）
  - `>`: 残り全セグメントにマッチ（`news.>` → `news.sports.football` ○）
- パターンマッチングアルゴリズム（セグメント分割と比較）

**実験ゴール**

```bash
# ターミナルB: SUB news.*
# ターミナルC: SUB news.>

# PUB news.sports Goal!
# → B と C 両方に届くこと

# PUB news.sports.football Touchdown!
# → C だけに届くこと（* は1セグメントなので B は届かない）
```

---

### Step 6: Replayバッファ

新しいサブスクライバーが接続したとき、そのトピックの直近N件のメッセージを即座に送ります。

**学習内容**
- リングバッファ（循環バッファ）の仕組みとGo実装
- サブスクライブ時のリプレイ: バッファを読んでから通常配信に切り替える
- バッファ参照中の書き込みとの排他制御（replay中に新メッセージが来たとき）

**実験ゴール**

```bash
# 先にメッセージをパブリッシュ（サブスクライバーがいない状態で）
nc localhost 4222
PUB news Message-1
PUB news Message-2
PUB news Message-3

# 後からサブスクライブ（-replay 3 オプション付き）
SUB news 3
# → Message-1, Message-2, Message-3 が即座に届いてから通常配信に移ること
```

## 学習を進める上でのアドバイス

1. **`nc` でプロトコルを手打ちして確認する**
   ブローカーに接続した `nc` でコマンドを手打ちするのが最も確実なデバッグ方法です。`SUB`, `PUB` を1行ずつ打って挙動を確認してください。

2. **goroutineリークに注意する**
   サブスクライバーが切断してもgoroutineが残り続けるケースが起きやすいです。`runtime.NumGoroutine()` を定期ログに出すと検出できます。

3. **`-race` フラグを常用する**
   サブスクライバーリストへの並列アクセスはデータ競合の温床です。`go run -race main.go` で開発中に検出する習慣をつけてください。

4. **NATSの仕様を参照する**
   このクラフトのプロトコルはNATSをシンプル化しています。NATSのプロトコル仕様（[nats.io/docs](https://docs.nats.io/reference/reference-protocols/nats)）を読むと設計の根拠が理解できます。

## 完走後の次のステップ

- **永続化**: メッセージをファイルやSQLiteに保存してブローカー再起動後もリプレイできるようにする
- **コンシューマーグループ**: 同一グループ内のサブスクライバーに1件ずつ排他配信する（kafka-cloneへの橋渡し）
- **ACK / At-least-once**: サブスクライバーがACKを返すまで再送する
- **クラスタリング**: 複数ブローカー間でメッセージを同期する
- **WebSocket対応**: TCP接続をWebSocketにしてブラウザからサブスクライブできるようにする（websocket-brokerとの融合）
