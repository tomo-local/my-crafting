# Step 3：エコーサーバー（前提知識）

---

## このステップで何が変わるか

Step 2 では受信のみでした。このステップでは受け取ったメッセージをそのまま送り返すエコーを実装します。「サーバーからクライアントへの送信」は受信時とフォーマットが異なる点（マスク不要）が重要です。

---

## サーバーからクライアントへのフレーム送信

RFC 6455 の規定:
- **クライアント → サーバー**: 必ずマスクする
- **サーバー → クライアント**: マスクしてはいけない

サーバーが送信するフレームの最小構成（テキスト、マスクなし）:

```
バイト 1: 0x81  → FIN=1, opcode=0x1 (text)
バイト 2: payload の長さ（125 以下の場合、MASK bit=0）
バイト 3〜: payload データ
```

`0x81` の内訳:
```
1000 0001
│       └── opcode = 0x1 (text)
└────────── FIN = 1
```

---

## フレーム長の組み立て（送信時）

payload 長に応じてバイト 2 以降が変わります（受信と同じルール）:

| payload 長 | バイト 2 | 後続 |
|---|---|---|
| 0〜125 | 長さをそのまま書く | なし |
| 126〜65535 | `0x7E`（126） | 次の 2 バイトに uint16 で書く |
| 65536〜 | `0x7F`（127） | 次の 8 バイトに uint64 で書く |

MASK bit は最上位ビットなので、サーバーからの送信では `byte2 = length`（最上位ビット 0）のまま使います。

---

## Close フレームの処理

クライアントが接続を閉じる際、opcode=0x8 の Close フレームを送ってきます。RFC 6455 の規定では、Close フレームを受け取ったらサーバーも Close フレームを返してから TCP を閉じます。

```
Client → Server: Close frame (opcode=0x8)
Server → Client: Close frame (opcode=0x8)  ← 返す
Server: conn.Close()
```

Close フレームの最小構成（payload なし）:
```
[0x88, 0x00]
```

`0x88` の内訳: `FIN=1, opcode=0x8 (close)`。

---

## 📌 まとめ: Step 3 のフロー

1. ループで `ReadFrame(conn)` を呼ぶ
2. opcode が Close (0x8) なら:
   - Close フレーム `[0x88, 0x00]` を返す
   - break
3. opcode が Text (0x1) または Binary (0x2) なら:
   - `FIN=1, opcode=受信と同じ, MASK=0, length` でヘッダーを組み立てる
   - payload をそのまま続けて書く
4. ループに戻る
