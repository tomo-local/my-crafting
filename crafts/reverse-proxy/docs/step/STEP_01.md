# Step 1：最小リクエスト転送（前提知識）

HTTPリクエストをそのまま別のサーバーに転送し、レスポンスをクライアントに返す——これがリバースプロキシの本質です。Step 1 ではロードバランシングもヘッダー操作もなし、1台のアップストリームへの最小転送だけを実装します。

---

## 1. プロキシが管理する2本のコネクション

通常のHTTPサーバーは「クライアントとの1本の接続」を管理します。プロキシはそれに加えて「アップストリームへの接続」を自分で確立します。

```
クライアント ──[接続A]── プロキシ ──[接続B]── アップストリーム
```

接続Aはクライアントが確立してきます（`Accept` で受け取る）。  
接続Bはプロキシが自分から確立しに行きます（`net.Dial`）。

---

## 2. 転送の中心：`io.Copy`

リクエスト・レスポンスのボディを転送するとき、自前でバッファを管理する必要はありません。Goの `io.Copy(dst, src)` がストリームを効率的にコピーしてくれます。

ただし、プロキシは**双方向**の転送が必要です。

```
クライアント → アップストリーム:  io.Copy(upstreamConn, clientConn)
アップストリーム → クライアント:  io.Copy(clientConn, upstreamConn)
```

この2つは**同時に走らせる**必要があります。片方が終わるまで待ってから反対方向を流すと、レスポンスが届かなくなります。

### goroutine による双方向コピー

```
goroutine 1: io.Copy(upstream, client)  // リクエストを流し続ける
goroutine 2: io.Copy(client, upstream)  // レスポンスを流し続ける
```

片方が `io.EOF`（接続クローズ）を受け取ったら、もう一方もそれに追随して終了する必要があります。

---

## 3. どちらが先に終わるか

HTTPの1リクエスト/1レスポンスの流れでは、通常は以下の順序です。

1. クライアントがリクエストを送り終える（`client → upstream` の `io.Copy` が終わる）
2. アップストリームがレスポンスを返し終える（`upstream → client` の `io.Copy` が終わる）

`io.Copy` は `src` の読み取りが `io.EOF` を返したときに終了します。接続を閉じると相手側の `io.Copy` も EOF を受け取って終了します。

---

## 4. アップストリームサーバーの準備

Step 1 では転送先として簡単なHTTPサーバーが必要です。`crafts/reverse-proxy/upstream/` に以下の機能を持つサーバーを作っておきます。

- 固定レスポンス（どのパスにも 200 を返す）
- どのサーバーから返ってきたか識別できるID（`-id` フラグ）
- レスポンスにサーバーIDを含める

---

## 📌 まとめ：Step 1 のフロー

1. `net.Listen` でプロキシのポートを開く
2. `Accept` でクライアントの接続を受け取る
3. `net.Dial` でアップストリームへの接続を確立する
4. goroutine 1: `io.Copy(upstream, client)` でリクエストを転送
5. goroutine 2: `io.Copy(client, upstream)` でレスポンスを転送
6. 両方の転送が完了したら両接続を `Close`
