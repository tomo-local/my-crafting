# リバースプロキシ自作に必要な知識の整理

リバースプロキシの自作は、大きく分けて**「プロキシの役割と設計」「HTTPプロトコルの転送ルール」「負荷分散アルゴリズム」「Go並行処理」**の4つの知識が必要です。

---

## 1. リバースプロキシとは何か

### 🔀 フォワードプロキシとの違い

どちらも「中継役」ですが、誰のために動くかが違います。

| | フォワードプロキシ | リバースプロキシ |
|---|---|---|
| 誰のため | クライアント側のため | サーバー側のため |
| クライアントの設定 | プロキシを明示的に指定する | プロキシの存在を知らなくてよい |
| 典型的な用途 | 社内からの外部アクセス制御、匿名化 | 負荷分散、SSL終端、キャッシュ |

リバースプロキシはクライアントから見ると「サーバーそのもの」に見えます。クライアントは `localhost:8080` に接続しているつもりでも、実際のレスポンスは `localhost:9001` から来ています。

### 🏗️ プロキシの基本フロー

```
クライアント            プロキシ                アップストリーム
    │                    │                         │
    │── リクエスト ──────>│                         │
    │                    │── リクエスト（転送） ──>│
    │                    │                         │
    │                    │<── レスポンス ──────────│
    │<── レスポンス ──────│                         │
    │                    │                         │
```

プロキシはこの中継を行うために、**2本のTCPコネクションを同時に管理**します。
- `クライアント ↔ プロキシ` の接続（受信用）
- `プロキシ ↔ アップストリーム` の接続（転送用）

## 2. HTTPプロトコルの転送ルール

### 📋 転送時に書き換えるべきヘッダー

リクエストをそのまま転送すると問題が起きるヘッダーがあります。

#### `Host` ヘッダー

クライアントが送ってくる `Host: localhost:8080` はプロキシのアドレスです。アップストリームに転送するときは、**アップストリームのアドレスに書き換える**必要があります。

```
クライアントの送信:  Host: localhost:8080
アップストリームへ:  Host: localhost:9001   ← 書き換える
```

#### `X-Forwarded-For`

アップストリームはプロキシのIPアドレスしか見えません。クライアントの本来のIPアドレスを伝えるために付加します。

```
X-Forwarded-For: 203.0.113.42
```

既存の `X-Forwarded-For` がある場合（多段プロキシ）は追記します。

```
X-Forwarded-For: 10.0.0.1, 203.0.113.42
```

#### `Via`

経路上のプロキシを記録するヘッダーです（RFC 9110）。

```
Via: 1.1 my-reverse-proxy
```

### ⚠️ hop-by-hop ヘッダーの除去

以下のヘッダーは「1つのTCP接続の中だけで有効」な意味を持ちます。アップストリームへの転送時に**削除する**のが正しい動作です。

| ヘッダー | 意味 |
|---|---|
| `Connection` | この接続をkeep-aliveにするかどうか |
| `Keep-Alive` | keep-aliveのパラメーター |
| `Transfer-Encoding` | チャンク転送の制御 |
| `Upgrade` | WebSocketへのアップグレード要求 |

## 3. 負荷分散アルゴリズム

### ⚖️ ラウンドロビン（Round Robin）

リクエストを順番にサーバーに割り当てます。最もシンプルで、すべてのリクエストが同じ処理コストを持つ場合に有効です。

```
サーバーリスト: [A, B, C]
1回目 → A
2回目 → B
3回目 → C
4回目 → A（ループ）
```

実装の核は「グローバルなカウンターを `atomic.AddUint64` でインクリメントし、サーバー台数で割った余りを使う」です。

### 🔗 Least Connections（最小コネクション）

現在アクティブなコネクション数が最も少ないサーバーにリクエストを送ります。処理時間が不均一（重いリクエストと軽いリクエストが混在）な場合に有効です。

```
サーバーA: アクティブ接続 3本
サーバーB: アクティブ接続 1本  ← ここに振り分ける
サーバーC: アクティブ接続 2本
```

実装では、リクエスト受付時にカウントを `+1`、レスポンス完了時（`defer` で確実に）`-1` します。

### 🔑 IP Hash

クライアントのIPアドレスをハッシュして、常に同じサーバーに振り分けます。ショッピングカートなどセッション情報がサーバーに保存される場合のセッションスティッキーに使います。

```
hash(client_ip) % サーバー台数 → サーバーインデックス
```

## 4. ヘルスチェックの仕組み

### 🏥 なぜ必要か

サーバーが突然落ちても、プロキシは即座には気づきません。ヘルスチェックなしだと落ちたサーバーにもリクエストを送り続け、クライアントにエラーが返り続けます。

### 🔄 アクティブヘルスチェック

バックグラウンドgoroutineが定期的にアップストリームの特定エンドポイント（`/health` など）にHTTPリクエストを送り、200が返れば「alive」、タイムアウトやエラーなら「dead」と判定します。

```
goroutine: 毎10秒ごとに /health をGET
  → 200 OK: alive フラグをセット
  → 接続エラー or タイムアウト: dead フラグをセット
```

### 🔒 状態管理の排他制御

ヘルスチェックgoroutineがalive/deadを書き込む一方、リクエスト転送goroutineが状態を読み取るため、`sync.RWMutex` で排他制御が必要です。

- 読み取り（振り分け判断）: `RLock` / `RUnlock`（複数goroutineが同時に読める）
- 書き込み（ヘルスチェック更新）: `Lock` / `Unlock`（排他）

## 5. コネクションプーリングの概念

### 🔌 TCPコネクション確立のコスト

TCPコネクションを新規に確立するたびに3ウェイハンドシェイクが発生します。ローカル環境でも数百マイクロ秒、WAN越しだと数十ミリ秒のオーバーヘッドです。

```
毎回確立する場合: リクエストごとに [SYN → SYN-ACK → ACK] が走る
プーリング使用時: 確立済み接続をキューから取り出して即使用
```

### 🏊 プールの基本設計

```
プール（idle接続のキュー）
    ┌──────────────────────────┐
    │ conn1 (idle)             │
    │ conn2 (idle)             │
    │ conn3 (idle)             │
    └──────────────────────────┘

リクエスト到着時:
  → キューにidle接続があれば取り出して使う
  → なければ新規接続を確立する

レスポンス完了時:
  → Connection: close なら接続を閉じる
  → そうでなければキューに戻す
```

## 6. 【実装上の罠】リバースプロキシで必ずハマるポイント

### 🛑 goroutineリークと双方向コピー

プロキシの転送は `io.Copy` を2方向（クライアント→アップ、アップ→クライアント）で同時に走らせます。どちらかの接続が切れたとき、もう一方の `io.Copy` がブロックしたまま終了しない問題が起きます。

**対策**: どちらかが終了したら両方の接続を閉じる。`sync.WaitGroup` とgoroutineを組み合わせるか、`context` でキャンセルを伝播させます。

### 🛑 `X-Forwarded-For` の偽装

クライアントが自分で `X-Forwarded-For: 10.0.0.1` を送ってくることができます。信頼できるプロキシのIPアドレスだけを元に判断するか、プロキシ側でヘッダーを上書きする方針を取る必要があります。

### 🛑 コネクションの再利用失敗

プールから取り出した接続が、アップストリームサーバー側でタイムアウト等により既に閉じられていることがあります（半開き接続）。書き込み時にエラーが返ったら、接続を破棄して新規接続に切り替えるリトライが必要です。
