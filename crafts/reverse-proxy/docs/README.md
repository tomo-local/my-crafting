# ゼロから構築するリバースプロキシ自作・全体学習プロセス

標準ライブラリの `httputil.ReverseProxy` を使わず、TCPソケットから直接リクエスト転送・負荷分散・ヘルスチェックを実装することで、リバースプロキシの仕組みを深く理解していくプロセスです。

---

## 全体ロードマップ

| Step | テーマ | ゴール |
|------|--------|--------|
| 1 | 最小リクエスト転送 | プロキシ経由で1台のアップストリームにリクエストが届き、レスポンスが返ること |
| 2 | プロキシヘッダーの付加 | `X-Forwarded-For` / `Via` / `Host` が正しくアップストリームに届くこと |
| 3 | ラウンドロビン負荷分散 | 複数アップストリームにリクエストが均等に振り分けられること |
| 4 | ヘルスチェック | 落ちたサーバーへのリクエストが自動で迂回されること |
| 5 | Least Connections | 最もアクティブコネクション数の少ないサーバーに振り分けられること |
| 6 | コネクションプーリング | アップストリームとのTCP接続を再利用し、レイテンシが下がること |

---

## 各ステップの詳細

### Step 1: 最小リクエスト転送

リバースプロキシの本質は「受け取ったリクエストを別のサーバーに転送し、レスポンスをそのまま返す中継役」です。まずこの最小形を作ります。

**学習内容**
- リバースプロキシとフォワードプロキシの違い
- プロキシの基本フロー: クライアント → プロキシ → アップストリーム → プロキシ → クライアント
- Goの `net.Dial` でアップストリームに接続する方法
- `io.Copy` による双方向ストリームのパイプ

**実験ゴール**

```bash
# アップストリームとして簡単なサーバーを別ポートで起動
go run upstream/main.go -port 9001

# プロキシを起動
go run main.go -upstream localhost:9001 -port 8080

# プロキシ経由でアクセス
curl http://localhost:8080/
# → アップストリームのレスポンスがそのまま返ること
```

---

### Step 2: プロキシヘッダーの付加

生のリクエストをそのまま転送すると、アップストリームはクライアントの本来のIPアドレスやプロキシの存在を知れません。HTTP標準で定められたヘッダーを付加します。

**学習内容**
- `X-Forwarded-For`: クライアントの元IPアドレスを伝えるヘッダー
- `Via`: 経路上のプロキシを記録するヘッダー
- `Host` ヘッダーの書き換え（クライアントのHostではなくアップストリームのHostにする理由）
- HTTPリクエストのヘッダー部分だけを書き換えてボディはそのまま転送するパース戦略

**実験ゴール**

```bash
# アップストリームでリクエストヘッダーをそのまま返すエコーサーバーを使う
curl http://localhost:8080/echo-headers
# → レスポンスに X-Forwarded-For: 127.0.0.1 が含まれること
# → Via: 1.1 reverse-proxy が含まれること
```

---

### Step 3: ラウンドロビン負荷分散

1台のアップストリームを複数台に拡張し、リクエストを均等に振り分けます。

**学習内容**
- ラウンドロビンアルゴリズムの仕組み（順番にインクリメントして剰余で選ぶ）
- `sync/atomic` を使ったgoroutine-safeなカウンター
- アップストリームリストの設定方法（フラグ or 設定ファイル）

**実験ゴール**

```bash
# 3台のアップストリームを起動
go run upstream/main.go -port 9001 -id upstream-1
go run upstream/main.go -port 9002 -id upstream-2
go run upstream/main.go -port 9003 -id upstream-3

# プロキシを複数アップストリームで起動
go run main.go -upstreams localhost:9001,localhost:9002,localhost:9003

# 3回アクセスして振り分けを確認
for i in 1 2 3; do curl -s http://localhost:8080/ ; done
# → upstream-1, upstream-2, upstream-3 の順で返ること
```

---

### Step 4: ヘルスチェック

アップストリームが落ちても自動でリクエストが迂回されるようにします。

**学習内容**
- バックグラウンドgoroutineによる定期ヘルスチェック（`time.Ticker`）
- サーバー状態（alive/dead）の管理と `sync.RWMutex` による排他制御
- フェイルオーバーのロジック（全台deadになった場合の503返却）
- ヘルスチェックエンドポイントの慣習（`/health`, `/ping` など）

**実験ゴール**

```bash
# アップストリームを2台起動
# → 2台にラウンドロビンされることを確認

# アップストリーム1台を停止
kill <upstream-1-pid>

# しばらく待ってから（ヘルスチェック間隔後）アクセス
curl http://localhost:8080/
# → 落ちたサーバーには振り分けられず、生きているサーバーだけに届くこと
```

---

### Step 5: Least Connections

リクエストの処理時間が不均一な場合、ラウンドロビンでは特定のサーバーに負荷が偏ります。アクティブコネクション数が最小のサーバーに振り分けます。

**学習内容**
- アクティブコネクション数のカウント手法（`sync/atomic` or `sync.Mutex` + map）
- リクエスト開始時にインクリメント、完了時にデクリメントするライフサイクル管理
- ラウンドロビンとの使い分け（レイテンシが均一ならRR、不均一ならLC）

**実験ゴール**

```bash
# レスポンスが遅いアップストリームと速いアップストリームを用意
go run upstream/main.go -port 9001 -delay 500ms
go run upstream/main.go -port 9002 -delay 0ms

# 並列アクセスしてどちらに多く振り分けられるか確認
for i in $(seq 1 10); do curl -s http://localhost:8080/ & done
wait
# → delay 0ms のサーバーにより多くのリクエストが割り当てられること
```

---

### Step 6: コネクションプーリング

毎回TCPコネクションを確立すると、3ウェイハンドシェイクのオーバーヘッドが発生します。確立済みの接続を再利用します。

**学習内容**
- TCP接続確立（3ウェイハンドシェイク）のコスト
- コネクションプールの基本設計: idle接続のキューと最大プールサイズ
- `HTTP/1.1 Connection: keep-alive` と接続再利用の関係
- idle接続の有効期限管理（使いすぎると古い接続でエラーが起きる）

**実験ゴール**

```bash
# プーリングなしとプーリングありのレイテンシ比較
# 100リクエストの合計時間を計測
time for i in $(seq 1 100); do curl -s http://localhost:8080/ > /dev/null; done
# → プーリングありの方が明らかに速いこと（接続確立コストの差）
```

## 学習を進める上でのアドバイス

1. **アップストリームサーバーを手元で動かす**
   本物のnginxやhttpdに向けるより、`crafts/reverse-proxy/upstream/` に簡単なエコーサーバーを作っておくと、ヘッダーの確認やdelayシミュレーションが自由にできます。

2. **Wiresharkや `tcpdump` でパケットを観察する**
   プロキシを経由したとき、TCPセッションが「クライアント↔プロキシ」「プロキシ↔アップストリーム」の2本になることをパケットキャプチャで確認すると理解が深まります。
   ```bash
   tcpdump -i lo -A port 8080 or port 9001
   ```

3. **goroutineリークに注意する**
   転送中にどちらかの接続が切れた場合、もう一方のgoroutineがブロックしたままになるケースがあります。`context` やチャネルを使った適切なキャンセル処理を意識してください。

4. **RFC を引く習慣をつける**
   `X-Forwarded-For` は [RFC 7239](https://www.rfc-editor.org/rfc/rfc7239) で標準化されています。`Via` は [RFC 9110 Section 7.6.3](https://www.rfc-editor.org/rfc/rfc9110#section-7.6.3) です。

## 完走後の次のステップ

- **HTTPS終端**: TLSをプロキシで終端してアップストリームにはHTTPで転送するSSLオフロード
- **IP Hash**: 同一クライアントIPを常に同じアップストリームに振り分けるセッションスティッキー
- **レート制限**: クライアントIPごとのリクエスト数制限
- **リトライ**: アップストリームが5xxを返した場合に別サーバーへ自動リトライ
- **設定ファイルの動的リロード**: SIGHUPでアップストリームリストを無停止で更新
