# ゼロから構築するHTTPサーバー自作・全体学習プロセス

標準ライブラリの高級なHTTPモジュールを使わず、OSの機能である「TCPソケット」だけを使って、ブラウザと通信できるWebサーバーを段階的に実装していく手順です。

---

## 全体ロードマップ

| Step | テーマ | ゴール |
|------|--------|--------|
| 1 | TCPソケットの基本 | `nc` でサーバーと文字のキャッチボールができる |
| 2 | 固定レスポンスの返却 | ブラウザに `Hello` が表示される |
| 3 | リクエストの解析 | パスに応じてレスポンスを変える |
| 4 | ボディの解析 | POST リクエストを受け取れる |
| 5 | 並行処理の導入 | 複数の接続を同時に捌ける |
| 6 | Keep-Alive | 1つの接続で複数リクエストを処理できる |

---

## 各ステップの詳細

### Step 1: TCPソケット通信の基礎

HTTPのことを忘れ、ネットワーク越しにデータを送受信する「土台」を作ります。

**学習内容**
- Goの `net` パッケージによる `Listen` / `Accept` の仕組み
- ブロッキングI/Oの挙動（`Accept` で処理が止まること）

**実験ゴール**

```bash
# サーバー起動後、別ターミナルから
nc localhost 8080
# → 入力した文字がサーバー側のコンソールに表示されれば成功
```

---

### Step 2: HTTP最小構成でのレスポンス

TCPの上にHTTPプロトコルのルールを適用します。

**学習内容**
- ブラウザが送ってくる生のリクエスト文字列を観察する
- HTTPレスポンスの最小構成: ステータスライン + ヘッダー + `\r\n\r\n` + ボディ

**実験ゴール**

```bash
curl -v http://localhost:8080
# → HTTP/1.1 200 OK が返ればブラウザでも表示される
```

最小限のレスポンス形式:

```
HTTP/1.1 200 OK\r\n
Content-Length: 13\r\n
\r\n
Hello, World!
```

---

### Step 3: リクエストラインのパースとルーティング

届いた文字列を分解し、パスに応じたレスポンスを返します。

**学習内容**
- リクエストの1行目（リクエストライン）を `\r\n` / スペースで分割する
- メソッド（`GET`/`POST`）とパス（`/`、`/about`）の抽出

**実験ゴール**
- `GET /` → 200 トップページ
- `GET /about` → 200 aboutページ
- それ以外 → 404 Not Found

---

### Step 4: リクエストボディの解析

POST リクエストのボディを読み取れるようにします。

**学習内容**
- `Content-Length` ヘッダーを読んでボディのバイト数を取得する
- ヘッダーとボディの境界（`\r\n\r\n`）を正確に検出する
- `Content-Length` 分だけ読むことが重要（それ以上読むとブロックする）

**実験ゴール**

```bash
curl -X POST http://localhost:8080/echo \
  -H "Content-Type: text/plain" \
  -d "hello"
# → 受け取ったボディをそのまま返す /echo エンドポイントを実装
```

---

### Step 5: 並行処理の導入（goroutine）

1つのリクエストを処理している間、他の接続が待たされないようにします。

> **注意**: Goの並行処理はOSスレッドではなく **goroutine** です。`go` キーワード1つで起動できる軽量な並行実行単位で、スレッドとは別物です。

**学習内容**
- `Accept` した瞬間、処理を goroutine に渡す
- goroutine を使わない場合との挙動の違い（シリアル実行 vs 並行実行）

**実装パターン**

```go
for {
    conn, _ := listener.Accept()
    go handleConn(conn) // goroutine に丸投げしてすぐ次の Accept へ
}
```

**実験ゴール**

```bash
# 複数タブまたは並列 curl で同時アクセスしても詰まらないことを確認
curl http://localhost:8080 & curl http://localhost:8080 & curl http://localhost:8080
```

---

### Step 6: Keep-Alive（持続的接続）

HTTP/1.1 のデフォルトである「1接続で複数リクエストを処理する」を実装します。

**学習内容**
- HTTP/1.1 では `Connection: close` を明示しない限り接続を維持するのがデフォルト
- 1つの接続ループの中でリクエストを繰り返し読む

**実験ゴール**

```bash
curl -v --http1.1 http://localhost:8080
# → Connection: keep-alive でも正常にレスポンスが返ること
```

## 学習を進める上でのアドバイス

1. **一気に作ろうとしないこと**
   最初から構造を綺麗にしようとすると詰まります。まず `main` に泥臭く書いて動かし、動いてから関数に切り出す順番で進めてください。

2. **`curl -v` を常に使う**
   ブラウザの挙動が怪しい時は `curl -v http://localhost:8080` で生のHTTPテキストを確認できます。`-i` でレスポンスヘッダーだけ見ることもできます。

3. **`nc` / `telnet` で生テキストを送る**
   パーサーのデバッグには、ブラウザより `nc` で手動送信する方が原因を特定しやすいです。

4. **RFC を引く習慣をつける**
   HTTPの仕様は [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110) にあります。「なぜ `\r\n` なのか」など、疑問が出たら一次情報を確認する習慣をつけると理解が深まります。

## 完走後の次のステップ

- **TypeScript/Node.js で再実装**: `net` モジュールを使って同じことをやる。イベントループとgoroutineの設計思想の違いを体感できます。
- **静的ファイルの配信**: `os.Open` でファイルを読んでレスポンスに流す
- **チャンキッド転送**: `Transfer-Encoding: chunked` の実装
- **HTTPS**: TLSハンドシェイクを実装してみる（難易度高）
