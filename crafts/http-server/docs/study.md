# ピュアソケットHTTPサーバー自作に必要な知識の整理

ソケット通信によるHTTPサーバーの自作は、大きく分けて**「OS・ネットワークの仕組み」「プロトコルの約束事」「サーバーの構造（並行処理）」**の3つの知識が必要です。

---

## 1. ネットワーク・ソケットの知識（土台）

プログラムがネットワークを介してデータをやり取りするための、OSが提供する仕組み（API）を理解する必要があります。

### 📡 TCP/IP 階層モデルにおける位置づけ
私たちが作るのは、**アプリケーション層（HTTP）**のデータを、OSの**トランスポート層（TCP）**の機能を使って届ける仕組みです。

* **IP（インターネットプロトコル）**: データを目的の「端末（PCやサーバー）」まで届ける役割。
* **TCP（トランスポートプロトコル）**: 届いたデータを「欠損なく、正しい順番で、信頼性高く」やり取りする役割。接続を開き（ハンドシェイク）、データを流し、接続を閉じる。
* **ポート番号**: 端末内の「どのプログラム（プロセス）」にデータを渡すかを識別する番号（例: HTTPは通常 `80`、開発時は `8080` など）。

### 🔌 ソケット（Socket）とは？
OSが用意している**「ネットワークの出入り口（蛇口）」**のようなものです。プログラムからは、ファイルと同じように「開く」「読み込む（Read）」「書き込む（Write）」「閉じる（Close）」という操作ができます。

### 🔄 サーバー側ソケットのライフサイクル
サーバーは以下の手順でソケットを扱い、クライアント（ブラウザ）からの接続を待ち受けます。

1. **Socket（作成）**: ソケットを生成する。
2. **Bind（紐付け）**: ソケットに「IPアドレス」と「ポート番号」を割り当てる。
3. **Listen（待ち受け開始）**: クライアントからの接続要求を受け付ける準備をする（行列の受付を開始するイメージ）。
4. **Accept（接続受け入れ）**: 接続してきたクライアントと1対1で通信するための**「新しいソケット」**を生成して返す。処理はここでクライアントが来るまでブロック（一時停止）します。
5. **Read / Write（データ送受信）**: 繋がったソケットを通じてデータを読み書きする。
6. **Close（切断）**: 通信が終わったらソケットを閉じる。

---

## 2. HTTPプロトコル（仕様・フォーマット）の知識

TCPソケットが繋がったら、その中を流れる「文字列のルール」がHTTPです。HTTPは非常にシンプルで、**「ただのプレーンテキスト（文字列）」**です。



### 📥 HTTPリクエストの構造（ブラウザ ➡️ サーバー）
ブラウザからソケットに送られてくる文字列は、以下のような形をしています。

```http
GET /index.html HTTP/1.1
Host: localhost:8080
User-Agent: Mozilla/5.0
Accept: text/html

（←ここには必ず空行「\r\n」が入る）
[リクエストボディ（POSTなどの場合のみデータが入る）]
```

# 自作サーバーのためのTCPプロトコル徹底解説

自作HTTPサーバーのコード（`Listen` や `Accept`、`Read` など）が、裏側でTCPのどんな仕組みを利用しているのかを整理します。

---

## 1. TCPの基本性質と役割（なぜ必要なのか）

ネットワークの基礎（IP）は、データを相手に投げっぱなしにする「手紙」のような仕組みです。途中でデータが消えたり、順番が前後したりします。
これだとWEBページがバグってしまうため、その上で動く**TCP（トランスポート層）**がすべてを解決します。

### 🛡️ TCPが保証してくれる3つのこと
* **データの信頼性**: 途中でデータが消えたら、自動的に「再送（送り直し）」してくれます。
* **順番の保証**: データがバラバラの順番で届いても、自動的に「正しい順番」に並び替えてくれます。
* **接続の確立**: データのやり取りを始める前に、お互いが通信可能か必ず確認します。

これらをすべて**OS（カーネル）が勝手にやってくれる**ため、私たちのプログラムは「データが壊れたらどうしよう」と心配せず、届いたデータをただ読み込むだけで済みます。

---

## 2. 実装に直結するTCPの3大ライフサイクル

自作サーバーで呼び出す関数が、TCPのどの動き（パケットの往復）に対応しているかを理解することが重要です。



### 🤝 ① 接続の確立：3ウェイ・ハンドシェイク（`Accept` の裏側）
TCPは、いきなりデータを送りません。通信を始める前に、必ず3回往復の挨拶を交わします。

1. **クライアント ➡️ サーバー**: 「もしもし、繋いでもいいですか？（SYN）」
2. **サーバー ➡️ クライアント**: 「いいですよ！こちらも準備OKです。そちらも大丈夫ですか？（SYN-ACK）」
3. **クライアント ➡️ サーバー**: 「確認しました！では通信を始めます！（ACK）」

> 💡 **自作サーバーでの意味**
> プログラムで `Accept()` を呼ぶと、この3回の挨拶が完了するまで処理が**一時停止（ブロック）**します。挨拶が終わると、OSから「個室ソケット」が返されて次の行へ進みます。

### 📦 ② データの送受信：確認応答とストリーム（`Read / Write` の裏側）
繋がった後はデータの送り合いですが、ここにTCP最大のクセがあります。

* **確認応答（ACK）**: データを送るたびに、相手から「無事に受け取った」という返事を待ちます。返事がなければOSが自動で再送します。
* **ストリーム（川の流れ）**: TCPはデータの「区切り」を管理しません。送られたデータはすべて、1文字ずつサラサラ流れる「1本のつながった川」として届きます。

### 🚪 ③ 接続の切断：4ウェイ・ハンドシェイク（`Close` の裏側）
通信が終わるときは、お互いに「もう送るデータは無いですね？」と確認し合ってから、通り道を消去します。

1. **サーバー ➡️ クライアント**: 「私はもう送るデータがありません（FIN）」
2. **クライアント ➡️ サーバー**: 「了解しました（ACK）」
3. **クライアント ➡️ サーバー**: 「私ももう送るデータはありません（FIN）」
4. **サーバー ➡️ クライアント**: 「了解しました（ACK）」

> 💡 **自作サーバーでの意味**
> プログラムで `Close()` を呼ぶことは、この「さようなら」の挨拶を開始することを意味します。これを行わないと、OSのメモリにゴミ（古い接続情報）が残り続け、サーバーが再起動できなくなります。

---

## 3. 【最重要】自作HTTPサーバーにおける「TCPの罠」

ソケットからHTTPサーバーを作る人が必ず直面する、実装上の最大の壁です。

### 🛑 「データの区切り」は自分で作らなければならない
クライアントが「Hello」という5文字を1回で送信したとしても、ネットワークの混雑状況によっては、サーバー側の `Read()` で以下のようにバラバラに届くことがあります。

* 1回目の `Read()` ➡️ `He` だけが読める
* 2回目の `Read()` ➡️ `llo` が遅れて読める

TCPは「ここまでが1回分のデータ」という目印をつけてくれません。
だからこそ、HTTPプロトコルでは**「ヘッダーの終わりには必ず `\r\n\r\n`（空行）を入れる」**、**「ボディの長さは `Content-Length: 18` のようにヘッダーに書く」**という厳格なテキストのルールを定めて、プログラム側でデータの区切りを判断できるようにしているのです。
