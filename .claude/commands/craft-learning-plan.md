# craft-learning-plan

`crafts/<name>/docs/` 配下に、学習プランのドキュメント群を作成するスキルです。

## ドキュメント構成

```
crafts/<name>/docs/
├── README.md          # 全体ロードマップ（学習の地図）
├── study.md           # ベース知識の整理（前提となる概念）
├── images/            # 図・ダイアグラム置き場
└── step/
    ├── STEP_01.md         # Step 1 の前提知識
    ├── STEP_01_impl.md    # Step 1 の実装ガイド
    ├── STEP_02.md
    ├── STEP_02_impl.md
    └── ...
```

---

## 各ファイルの役割と書き方

### `docs/README.md` — 全体ロードマップ

**目的**: 「何をどの順番で作るか」を一覧で把握できる地図。

**必須セクション**:

1. **タイトルと一言説明** — このクラフトで何を作るか
2. **全体ロードマップ表** — `| Step | テーマ | ゴール |` の3列テーブル
3. **各ステップの詳細** — Step ごとに以下を書く:
   - 学習内容（箇条書き）
   - 実験ゴール（`curl` や `nc` で確認できる具体的なコマンド）
4. **学習を進める上でのアドバイス** — 詰まりやすいポイント、デバッグ方法、参照先
5. **完走後の次のステップ** — 発展課題

**書き方の指針**:
- ゴールは「コマンドを打てば動くことを確認できる」レベルまで具体化する
- 学習内容は「なぜそれを学ぶか」が伝わる粒度で書く
- Step 数は 5〜8 が目安（多すぎると挫折する）

---

### `docs/study.md` — ベース知識の整理

**目的**: 実装を始める前に「頭に入れておくべき概念」を体系的にまとめる。実装から独立した参照ドキュメント。

**必須セクション**:

1. **このクラフトに必要な知識の全体像** — 大分類（例: ネットワーク知識 / プロトコル仕様 / サーバー構造）
2. **各大分類の詳細説明** — 以下を含む:
   - 概念の定義（初見でも理解できる言葉で）
   - プログラムの動作との対応（「`Accept()` を呼ぶとOSで何が起きるか」など）
   - 図・アスキーアートがあると効果的
3. **実装上の罠・注意点** — 「知らないと必ずハマる」内容を最後にまとめる

**書き方の指針**:
- 実装コードは書かない（`study.md` は概念の説明、実装は `impl.md` に分ける）
- 絵文字アイコン（📡 🔌 ⚠️ など）を使って視覚的にセクションを区別する
- 「プログラム側の意味」と「OS/ネットワーク側の動き」を並べて説明すると理解が深まる

---

### `docs/step/STEP_XX.md` — Step の前提知識

**目的**: そのステップを実装するために必要な「理論・仕様の理解」。実装手順は書かない。

**必須セクション**:

1. **タイトル** — `# Step N：〇〇（前提知識）`
2. **このステップで何が変わるか** — 前ステップとの差分を1〜2文で説明
3. **核となる概念の説明** — 仕様・プロトコル・アルゴリズムをコード無しで解説
4. **図や例示** — HTTPヘッダー例、パケットの流れ図、テーブルなど
5. **📌 まとめ: Step N のフロー** — 実装するロジックの順序を箇条書きで

**書き方の指針**:
- このファイルを読んだ後、`impl.md` を見て「あとは書くだけ」と感じるレベルまで理論を整理する
- 「なぜそう設計されているか」（RFC の意図など）に触れると理解が深まる
- 実装コードの断片を書く場合は、あくまで概念の例示として最小限に留める

---

### `docs/step/STEP_XX_impl.md` — Step の実装ガイド

**目的**: 「何をどの関数にどの順番で書くか」の具体的な指示書。理論の説明は最小限。

**必須セクション**:

1. **タイトル** — `# Step N 実装ガイド：〇〇`
2. **ゴール** — 確認コマンドと期待する出力（コードブロックで明示）
3. **変更するファイル** — ディレクトリツリーで示す
4. **関数ごとの実装手順** — 各関数について:
   - 「内部でやること（順番どおり）」を箇条書き
   - 要所にコードスニペット（部分的でよい）
   - 前ステップからの差分がある場合は `> **Step N との差分**` で明示
5. **実装の確認手順** — ビルド確認 → 動作確認の順で `bash` コードブロックを使う
6. **よくあるハマりポイント** — `| 症状 | 原因 | 対処 |` の3列テーブル

**書き方の指針**:
- 実装の手順は「順番どおり」に書く。依存関係が崩れると動かないので順序が重要
- コードスニペットは「答えをそのまま渡す」のではなく、「迷う箇所だけ示す」粒度が適切
- 確認手順は「ビルドが通ること」を最初に置く（ランタイムエラーより先にコンパイルエラーを消す）
- ハマりポイントは実際に起こりうる症状ベースで書く（「〇〇を忘れた場合」ではなく「〇〇が起きる」）

---

## 実行手順

引数 `$ARGUMENTS` からクラフト名とステップ構成を読み取り、以下の順で作成してください：

1. `docs/README.md` — 全体ロードマップ
2. `docs/study.md` — ベース知識
3. `docs/step/STEP_01.md` 〜 `STEP_NN.md` — 各ステップの前提知識
4. `docs/step/STEP_01_impl.md` 〜 `STEP_NN_impl.md` — 各ステップの実装ガイド

引数が「既存の README.md を確認して作成」のような指示の場合は、対象ディレクトリの README.md を先に読んでからゴールとステップ数を決める。

引数: $ARGUMENTS
